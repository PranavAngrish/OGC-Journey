Step 02 · Learning MCP

Studying the Model Context Protocol

MCP (Model Context Protocol) is an open protocol developed by Anthropic that lets LLMs interact with external tools and services in a structured, standardised way. Instead of every application building custom LLM integrations, MCP provides a common language. An LLM can discover tools, understand what parameters they need, call them, and get structured responses back.

I spent time understanding the core concepts: how MCP servers expose tools, how MCP clients (LLMs) discover and call those tools, and how the protocol handles tool schemas, responses, and errors. The key insight was that MCP is not an API wrapper — it is a formal contract between natural language intent and executable code.

Understanding MCP

What is the Model Context Protocol?

MCP is a standardised protocol that lets LLMs interact with external systems — databases, APIs, file systems — in a consistent, structured way. Think of it as the USB standard for LLM integrations.

⬡
Tool Definition
Every external capability is described as a named tool with a typed parameter schema. The LLM reads the description and decides when to call it based on user intent.
⬡
Structured Discovery
MCP servers advertise their available tools. LLMs discover capabilities at runtime — the same LLM can work with any MCP server without retraining.
⬡
Protocol Separation
MCP decouples the LLM from the underlying system. When an API changes, only the MCP mapping updates — every application built on it benefits automatically.
⬡
Modular by Design
Each tool is independently defined and reusable. A tool for querying OGC API – Features works regardless of which LLM, which server, or which application uses it.
mcp_tool_example.py — What an MCP tool looks like in code
# MCP tool definition for OGC API – Features
# The LLM reads the description and decides when to call this

@mcp.tool()
def get_features(
    collection: str,
    bbox: list[float] | None = None,
    filter: str | None = None,
    limit: int = 100
) -> dict:
    """
    Fetches geospatial features from an OGC API – Features server.
    Use when the user wants to find, query, or retrieve geographic
    features like buildings, roads, parks, or boundaries.

    Args:
      collection: Name of the feature collection (e.g. 'buildings')
      bbox: Optional bounding box [minLon, minLat, maxLon, maxLat]
      filter: Optional CQL2 filter string for attribute queries
      limit: Maximum number of features to return
    """
    # Proxy translates this to real OGC API HTTP call
    return ogc_proxy.get(f"/collections/{collection}/items", params=...)

The GSoC project is about creating this kind of tool definition — formally, extensibly, and machine-readably — for the entire suite of OGC APIs. So any developer building an LLM application that needs geospatial data can use the spec instead of figuring out OGC API integration from scratch.

OGC API Standards

The Four APIs that Matter

The Open Geospatial Consortium defines how geospatial data is served over HTTP. Each API type serves a different kind of geospatial need. Together they cover almost every geospatial workflow a non-expert might want to run.

OGC API – Features
Vector Geospatial Data
Query geographic features — buildings, roads, parks, boundaries — as GeoJSON. Supports spatial filters (bbox, INTERSECTS) and attribute filters (CQL2). This is where most urban planning data lives.
OGC API – Records
Metadata Catalogue
Discover what datasets and services exist. Like a search engine for geospatial data — find datasets by keyword, location, or time period. The entry point for any data-driven workflow.
OGC API – EDR
Environmental Data
Query environmental time-series data — temperature, rainfall, air quality, wind — at a point, within an area, or along a trajectory. Powers climate and environmental analysis.
OGC API – Processes
Geoprocessing Execution
Execute geospatial analysis processes remotely. Submit a job, monitor progress, retrieve results. This is the API I built for the coding challenge — buffer operations, zonal statistics, and more.

Key Technical Decisions

▸
JWT from scratch — Implemented HMAC-SHA256 JWT using Python standard library only (hmac, hashlib, base64). No external JWT dependency. Tokens are fully spec-compliant and decodable by jwt.io.
▸
Network isolation — pygeoapi and auth service are never exposed to the host. Only nginx listens on port 80. The internal ogc-internal bridge network is the only communication path.
▸
auth_request pattern — nginx validates every JWT via an internal subrequest to the auth service before proxying to pygeoapi. If auth fails, pygeoapi never sees the request.
▸
Both execution modes — Synchronous (HTTP 200, immediate result) and asynchronous (Prefer: respond-async, HTTP 201 + jobID) execution verified end-to-end.
▸
Geodesic buffer — Projects WGS84 → EPSG:3857 (metric), applies buffer, reprojects back. A naive buffer in degrees would produce an oval, not a circle.
▸
Frontend UI — Interactive React + Leaflet map at /ui with API explorer, job management, request log, and in-app documentation. No build step — served as a single HTML file.

What's Next

The GSoC Project Ahead

The coding challenge built the foundation — the OGC API backend that needs to be talked to. The GSoC project builds everything that sits on top of it: the mapping spec and the reference implementation.

01
MCP Mapping Specification
A formal, machine-readable JSON schema that describes OGC API operations as MCP tools. Modular — each OGC API type is independently usable. Extensible — new API types can be added without redesigning existing modules.
02
Reference Implementation
A Python library or proxy service that implements the mapping spec. Takes MCP tool calls from any LLM and translates them to real OGC API HTTP requests — handling auth, CRS, pagination, format negotiation transparently.
03
Showcase Demo
A working demonstration of the full system — an urban planner types "conduct a cool spot analysis based on my new parks" and the LLM, guided by the MCP mapping, executes the full multi-step geospatial workflow and returns a visualised result.

Understanding MCP

What is the Model Context Protocol?

MCP is a standardised protocol that lets LLMs interact with external systems — databases, APIs, file systems — in a consistent, structured way. Think of it as the USB standard for LLM integrations.

⬡
Tool Definition
Every external capability is described as a named tool with a typed parameter schema. The LLM reads the description and decides when to call it based on user intent — no explicit routing code needed.
⬡
Structured Discovery
MCP servers advertise their available tools at runtime. The same LLM can work with any MCP server without retraining — it just reads the tool list and reasons about which one to call.
⬡
Protocol Separation
MCP decouples the LLM from the underlying system. When an API changes, only the MCP mapping updates — every application built on top of it benefits automatically.
⬡
Modular by Design
Each tool is independently defined and reusable. A tool for querying OGC API – Features works regardless of which LLM, which server, or which application uses it.
ogc_mcp_server.py — FastMCP tool definition (actual code from the prototype)
# Each tool is one decorated function.
# The docstring IS the tool description the LLM reads.
# Type hints ARE the input schema FastMCP generates.

@mcp.tool()
def execute_buffer(latitude: float, longitude: float, distance: float) -> str:
    """Creates a circular buffer polygon around a coordinate point.
    Use when the user wants to find things within a distance of a location,
    create a zone around a point, or draw a radius around a place.
    Args:
      latitude: Latitude in decimal degrees
      longitude: Longitude in decimal degrees
      distance: Buffer radius in metres
    """
    # Auth handled by proxy — LLM never sees credentials
    response = requests.post(
        f"{OGC_BASE_URL}/processes/buffer/execution",
        headers=auth_headers(),
        json={"inputs": {"latitude": latitude, "longitude": longitude, "distance": distance}}
    )
    return summarise_response(response.json())
OGC API Standards

The Four APIs that Matter

The Open Geospatial Consortium defines how geospatial data is served over HTTP. Each API type serves a different kind of need. Together they cover almost every geospatial workflow a non-expert might want to run.

OGC API – Features
Vector Geospatial Data
Query geographic features — buildings, roads, parks, boundaries — as GeoJSON. Supports spatial filters (bbox, INTERSECTS) and attribute filters (CQL2). This is where most urban planning data lives.
OGC API – Records
Metadata Catalogue
Discover what datasets and services exist. Like a search engine for geospatial data — find datasets by keyword, location, or time period. The entry point for any data-driven workflow.
OGC API – EDR
Environmental Data
Query environmental time-series data — temperature, rainfall, air quality, wind — at a point, within an area, or along a trajectory. Powers climate and environmental analysis.
OGC API – Processes
Geoprocessing Execution
Execute geospatial analysis remotely. Submit a job, monitor progress, retrieve results. This is the API built for the coding challenge — buffer operations, zonal statistics, full async lifecycle.

Key Technical Decisions

▸
JWT from scratch — Implemented HMAC-SHA256 JWT using Python standard library only. No external JWT dependency. Tokens are fully spec-compliant and decodable by jwt.io.
▸
Network isolation — pygeoapi and auth service are never exposed to the host. Only nginx listens on port 80. The internal ogc-internal bridge network is the only communication path.
▸
auth_request pattern — nginx validates every JWT via an internal subrequest to the auth service before proxying to pygeoapi. If auth fails, pygeoapi never sees the request.
▸
Both execution modes — Synchronous (HTTP 200, immediate result) and asynchronous (Prefer: respond-async, HTTP 201 + jobID) execution verified end-to-end.
▸
Geodesic buffer — Projects WGS84 → EPSG:3857 (metric CRS), applies buffer, reprojects back. A naive buffer in degrees produces an oval, not a circle.
▸
Frontend UI at /ui — Interactive React + Leaflet map with API explorer, job management, request log, and in-app documentation. No build step — single self-contained HTML file.

What This Proves

▸
Interoperability — A single LLM conversation chains three independent OGC API servers (localhost pygeoapi, demo.pygeoapi.io, demo.pycsw.org) through one unified interface.
▸
Reduced complexity — The LLM calls tools in plain English. All HTTP, auth, format negotiation, and response summarisation is hidden inside the proxy layer.
▸
Intelligent behaviour — The system automatically broadens searches, chains context across turns, and interprets results with domain awareness — not just raw API responses.
▸
Auth transparency — The MCP server obtains JWT tokens and injects them on every request to the authenticated pygeoapi instance. The LLM never sees credentials.

What the spec defines for every tool

▸
natural_language_triggers — example phrases that guide the LLM toward this tool. Serve as documentation and automated test cases.
▸
http_mapping — exact HTTP method, URL path template, which parameters are path vs query vs body.
▸
input_schema — every parameter with type, required flag, description, default, and validation rules.
▸
output_schema + summary_fields — what the response contains and which fields to extract before passing to the LLM.
▸
prerequisites + followup_tools — which tools should be called before and after, enabling intelligent workflow chaining.
▸
error_handling — every HTTP error code translated to plain English with a suggested LLM action.
ogc-mcp-mapping.json — excerpt showing spec structure
{
  "id": "features_get_feature_by_id",
  "module": "features",
  "description": "Gets the complete geometry of a single feature...",
  "natural_language_triggers": [
    "get the geometry of X", "I need the boundary of X"
  ],
  "prerequisites": ["features_get_items"],
  "http_mapping": {
    "method": "GET",
    "path_template": "/collections/{collection_id}/items/{feature_id}"
  },
  "output_schema": {
    "summary_fields": ["id", "geometry", "properties"],
    "followup_tools": ["processes_execute_sync"]
  }
}
